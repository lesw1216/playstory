# 엑셀 생성 (비동기)

> 생성일: 2026-06-02
> 브랜치: feat/excel-generation (예정, main에서 분기)

## 목표

`POST /api/excel-jobs` 요청 시 즉시 jobId+PENDING을 응답하고, 백그라운드에서 `orders`(10만건)를 .xlsx로 생성하며 상태를 PENDING→PROCESSING→DONE/FAILED로 전이시킨다. (설계 §2·§3)

## 결정 사항

- 엑셀 라이브러리: Apache POI `poi-ooxml:5.3.0` — SXSSF 스트리밍(.xlsx)
- 비동기: `@Async` + 전용 `ThreadPoolTaskExecutor`
- 실패 원인: 로그만(미영속). status=FAILED만 DB 반영 (error_log/컬럼 추가 없음)

## 비동기 설계

commit-before-async race(트랜잭션 commit 전 비동기 조회 시 job 못 찾음)를 `@TransactionalEventListener(AFTER_COMMIT)` + `@Async`로 회피한다.

```
[HTTP] ExcelJobService.create() @Transactional
  ├─ save(ExcelJob.pending())                 // PENDING insert
  ├─ publishEvent(ExcelJobRequestedEvent(jobId))
  └─ return ExcelJobResponse(jobId, PENDING)  // 즉시 응답
         │ (트랜잭션 commit)
         ▼ AFTER_COMMIT
[excelExecutor] ExcelGenerator.onExcelJobRequested(event)
  @Async("excelExecutor") @TransactionalEventListener(AFTER_COMMIT)
  ├─ excelJobService.markProcessing(jobId)        // 짧은 @Transactional
  ├─ Path f = orderExcelWriter.write(jobId)       // 트랜잭션 밖, 청크 스트리밍
  ├─ 성공 → excelJobService.markDone(jobId, f)
  └─ 예외 → log.error + excelJobService.markFailed(jobId)
```

- 의존성 사이클 없음: `ExcelJobService → ApplicationEventPublisher`(역참조 X), `ExcelGenerator → ExcelJobService(mark*) + OrderExcelWriter`.
- 스레드풀: core 1 / max 2 / queueCapacity 50 / CallerRunsPolicy, thread-name `excel-` (0.5 CPU 보호).

## 대용량(10만건) 전략

- 읽기: keyset 페이징(`id > lastId ORDER BY id LIMIT chunk`) 청크 반복 — 전체 List 적재 X, deep-offset 비용 X
- 쓰기: SXSSF window=100 → 메모리 100행만 유지, 나머지 temp flush. 10만건 단일 시트 OK
- 트랜잭션: 파일 생성 동안 장기 트랜잭션 금지. 상태 전이만 짧은 @Transactional. `workbook.dispose()`로 temp 정리

## 엑셀 컬럼

id / user_name / product_name / category / amount / status / order_date (Order 엔티티 그대로, order_date는 문자열 포맷)
