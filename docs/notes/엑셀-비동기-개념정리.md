# 엑셀 생성 비동기 — 개념 정리 (Q&A)

엑셀 생성 비동기 구현에서 헷갈리기 쉬운 핵심 개념을 Q&A로 정리한다.

관련 코드
- `backend/.../api/exceljob/service/ExcelJobService.java` — `create()` + 상태 전이
- `backend/.../api/exceljob/service/ExcelGenerator.java` — 비동기 워커
- `backend/.../api/exceljob/event/ExcelJobRequestedEvent.java` — 이벤트
- `backend/.../config/AsyncConfig.java` — `@EnableAsync` + `excelExecutor` 풀

---

## 1. "커밋"은 git commit이 아니다 — 트랜잭션 commit

`@Transactional` 메서드는 하나의 **DB 트랜잭션**으로 묶인다.

- `create()` 안에서 `excelJobRepository.save(...)`를 호출해도 그 시점엔 DB에 **확정 저장이 안 된다.** 트랜잭션이 열려 있어 다른 커넥션에서는 그 PENDING row가 안 보인다.
- `create()`가 정상 종료될 때 Spring이 트랜잭션을 **commit** → 이때 INSERT가 DB에 확정되고 다른 커넥션(=다른 스레드)에서도 조회 가능해진다.

| 용어 | 의미 |
|------|------|
| git commit | 코드 변경 이력 저장 (이 맥락과 무관) |
| **트랜잭션 commit** | DB INSERT/UPDATE를 **확정**하는 것. `@Transactional` 메서드 정상 종료 시 발생 |

## 2. `create()`의 `eventPublisher` 역할

```java
eventPublisher.publishEvent(new ExcelJobRequestedEvent(job.getId()));
```

- `ApplicationEventPublisher`는 Spring이 기본 제공하는 **이벤트 발행기**(생성자 주입으로 자동 주입, 별도 설정 불필요).
- 위 한 줄은 **"jobId가 X인 엑셀 생성 요청이 발생했다"는 사건을 Spring에 알리는** 것. 누가 받을지는 신경 쓰지 않는다.

### 왜 `excelGenerator.generate()`를 직접 호출하지 않았나

**(1) 트랜잭션 commit 타이밍 제어 (가장 중요)**
`@TransactionalEventListener(AFTER_COMMIT)`은 트랜잭션 commit이 끝난 직후에만 리스너를 실행시킨다. 직접 호출은 이 보장이 없어, PENDING이 DB에 확정되기 전 비동기 스레드가 조회하면 race가 난다.

**(2) 의존성 사이클 제거 (디커플링)**
```
직접 호출:  ExcelJobService → ExcelGenerator → ExcelJobService(mark*)   ← 순환 의존!
이벤트:     ExcelJobService → ApplicationEventPublisher (Spring 코어)
            ExcelGenerator  → ExcelJobService(mark*)
            (ExcelJobService는 ExcelGenerator를 전혀 모름 → 사이클 없음)
```

## 3. `@Async("excelExecutor")`

**"이 메서드를 호출 스레드가 아니라 별도 스레드에서 실행하라"**는 표시.

- `@Async`가 없으면 메서드는 호출한 스레드에서 동기 실행된다.
- `@Async`가 붙으면 Spring이 빈을 프록시로 감싸 호출 즉시 작업을 **스레드풀에 넘기고 바로 리턴**한다. 본문은 풀의 워커 스레드에서 실행.
- 괄호 안 `"excelExecutor"`는 **어느 스레드풀을 쓸지** 지정하는 bean 이름.

```java
@Bean("excelExecutor")          // ← 이 이름과 매칭
public Executor excelExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(1);
    executor.setMaxPoolSize(2);   // 컨테이너 0.5 CPU 보호용으로 작게
    executor.setQueueCapacity(50);
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    ...
}
```

- 이름을 안 주면 Spring 기본 executor(무제한 스레드 `SimpleAsyncTaskExecutor`)를 써서 대량 작업 시 위험 → **전용 풀을 명시**.
- 전제: `AsyncConfig`에 `@EnableAsync`가 있어야 `@Async`가 동작한다.

## 4. `@TransactionalEventListener` + 리스너는 파라미터 타입으로 매칭

```java
@Async("excelExecutor")
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onExcelJobRequested(ExcelJobRequestedEvent event) { ... }
//                              └─ 이 파라미터 타입이 매칭 기준
```

- `publishEvent(객체)`가 호출되면 Spring은 **그 객체 타입과 호환되는 파라미터 타입**을 가진 리스너를 찾아 호출한다.
- `publishEvent(new ExcelJobRequestedEvent(...))` → 파라미터가 `ExcelJobRequestedEvent`인 이 메서드가 매칭.
- **메서드 이름은 매칭과 무관**하다. 기준은 오직 파라미터 타입. 같은 타입 리스너가 여러 개면 모두 호출(순서는 `@Order`).
- `AFTER_COMMIT`이라 비동기 스레드는 항상 트랜잭션 commit된 job을 조회 → 조회 race 없음.

## 5. 세 어노테이션 = (어디서 / 언제 / 무엇에 반응)

```
@Async("excelExecutor")                            → excel- 풀의 별도 스레드에서   (어디서)
@TransactionalEventListener(phase = AFTER_COMMIT)   → 트랜잭션 commit 직후에        (언제)
public void onExcelJobRequested(ExcelJobRequestedEvent e) → 이 타입 이벤트가 오면 실행 (무엇에)
```

## 6. 전체 흐름 + race 회피

```
[HTTP 스레드] ExcelJobService.create()  @Transactional
   ├─ save(ExcelJob.pending())                 // PENDING insert (아직 미확정)
   ├─ publishEvent(ExcelJobRequestedEvent(jobId))
   └─ return PENDING                           // 즉시 응답
            │ 트랜잭션 commit                   // 여기서 PENDING이 DB에 확정
            ▼ AFTER_COMMIT 시점에만 발화
[excelExecutor 스레드] ExcelGenerator.onExcelJobRequested(event)
   ├─ markProcessing(jobId)                     // 짧은 @Transactional
   ├─ file = orderExcelWriter.write(jobId)      // 트랜잭션 밖, 청크 스트리밍
   ├─ 성공 → markDone(jobId, file)
   └─ 예외 → log.error + markFailed(jobId)
```

핵심: PENDING job이 **트랜잭션 commit으로 DB에 확정된 뒤**에만 비동기 워커가 시작되므로, 별도 커넥션을 쓰는 워커가 job을 못 찾는 **commit-before-async race**가 발생하지 않는다.
