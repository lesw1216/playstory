schema.sql 기반 JPA 엔티티 생성

2026-06-01 20:24

orders, excel_export_jobs 두 테이블에 대응하는 JPA 엔티티와 상태 enum을 생성했다.

작업 요청 요약

  • backend/src/main/resources/db/schema.sql 의 테이블 2개를 보고 엔티티 생성.
  • 기존 백엔드 규칙(dto-layer 패키지 구조, coding-convention의 enum·Rich Domain) 준수.

사전 계획 · 핵심 결정

  • ddl-auto: validate → 스키마와 정확히 일치하도록 타입·nullable·길이 매핑.
  • id integer → 엔티티 Integer + IDENTITY (Long=bigint 불일치 방지).
  • 클래스↔테이블명 불일치 → @Table(name=...) 명시 (orders / excel_export_jobs).
  • orders.status: OrderStatus enum + @Enumerated(STRING), Converter 없음 → 저장값은 대문자 enum 이름(추후 더미데이터도 대문자로 생성 전제).
  • BaseEntity 미사용: 두 테이블에 audit 컬럼 없음 + validate 모드라 추가 컬럼 금지.

생성 파일

파일 유형 내용
api/order/model/Order.java + orders 매핑. id/userName/productName/category/amount/status/orderDate. 행위 메서드 없음(read-only 원본)
api/order/model/OrderStatus.java + CONFIRMED / CANCELLED / PENDING
api/exceljob/model/ExcelJob.java + excel_export_jobs 매핑. 상태 전이 메서드 pending()·startProcessing()·complete()·fail() 포함
api/exceljob/model/ExcelJobStatus.java + PENDING / PROCESSING / DONE / FAILED

패키지 base: com.playstory.backend.api.{domain}.model

검증 결과

  • ./gradlew compileJava → BUILD SUCCESSFUL (컴파일 통과).
  • 스키마 정합성(validate)은 MySQL 기동 + bootRun 시 확인됨 — 이번엔 컴파일까지만 수행.

알려진 제약 · follow-up

  • 스키마 nullable 정합화(해결됨): excel_export_jobs 의 started_at·ended_at·file_path 를 NULL 허용으로 조정해, PENDING은 시작/종료/파일이 없고 PROCESSING은 종료/파일이 없는 비동기 흐름이 가능해졌다. 엔티티의 nullable 매핑과도 일치.
  • orders.status 대소문자: Converter 없는 enum이므로 DB 저장값은 대문자. 더미데이터 생성 시 대문자로 맞춰야 조회가 깨지지 않음. 소문자 저장이 필수라면 AttributeConverter 추가 필요.
  • Repository / Service / DTO 는 이번 범위 밖(엔티티만 생성).
